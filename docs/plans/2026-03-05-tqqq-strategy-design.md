# TQQQ 전략 고도화 설계서 (Design)

## 1) 목표
- **1차 완료:** Pine 로직 동치 백테스트 구현
- **2차 완료:** 성과 개선 실험 프레임 구축
- **3차 완료:** 운영 자동화(신호/알림) + 웹 대시보드

## 2) 확정된 핵심 의사결정
- 동치 기준: 일자별 `weight` 오차 **±0.1% 이내**
- 체결 가정: **당일 RTH 종가 체결(16:00 ET 기준)**
- 비용 가정: 편도 5 bps (수수료 3 + 슬리피지 2), 왕복 10 bps
- 세후 가정: 해외주식 양도차익 **연 250만원 공제 후 22%**, TQQQ 단독 모델
- 벤치마크: TQQQ B&H, QQQ B&H
- 데이터 정책:
  - 신호 계산: yfinance adjusted(dividends)
  - 체결/성과: stooq bulk raw
  - 환율: yfinance `KRW=X`
- 2차 실험: 구조 변경 자유 허용 + **OOS 성과 유지율 70% 이상** 게이트
- 3차 운영: 자동 주문 없음(신호/알림 자동화, 수동 주문)
- 배포: 로컬 우선 → 안정화 후 클라우드

## 3) 기간 정책
### 공식(Primary)
- 1차 비교 공식 구간: **2011-06-23 ~ 2026-01-30**

### 보조 스트레스(Secondary)
- 프리라이브 합성: **2000-01-03 ~ 2010-02-10**
- 이벤트 구간:
  - 닷컴 붕괴: 2000-03-24 ~ 2002-10-09
  - 금융위기: 2007-10-09 ~ 2009-03-09
  - 위기 후 반등: 2009-03-10 ~ 2010-02-10
- 롤링: 1Y / 3Y
- 주의: Secondary 결과는 공식 성과표와 분리

## 4) 아키텍처(2안 + 3안 전환 가능형)
1. `data_ingest`: stooq/yfinance 수집
2. `data_canonical`: 표준 스키마/품질검사/Parquet + DuckDB
3. `signal_engine`: Pine 동치 상태기계 산출
4. `backtest_engine`: 체결/비용/세후/벤치마크
5. `research_lab`: WFO/OOS/민감도/리더보드
6. `ops_webapp`: 운영 대시보드 + 일일 알림

## 5) 전략 우선순위(상태기계)
1. 변동성 락(20일 변동성)
2. SPY 200MA 필터
3. 고비중 구간 원금보호 손절
4. 손절 후 RSI 재진입 잠금
5. 기본 추세 비중(100/10/0)
6. 과열 구간 단계 감량
7. TP10(+10% 도달 시 100→95)

## 6) 데이터 계약 & 정합성
- 기준 시간대: `America/New_York`
- 거래 세션: RTH 종가 이벤트 기준
- canonical 스키마 필수 필드:
  - `date, symbol, open, high, low, close, adj_close, volume, source, tz, session_type, is_trading_day`
- 정합성 체크:
  - 결측/중복/역정렬/휴장일 불일치
  - 원천 불일치 시 경고 및 보수적 skip
- 실행 재현성:
  - `run_id`, 데이터 해시, 전략 버전, 코드 버전 저장

## 7) 검증 게이트 (1차 완료 기준)
- Pine export 대비 일자별 `weight` 절대오차 <= 0.001
- mismatch 리포트 자동 생성(일자, expected, actual, delta, reason)
- 성과 지표 산출:
  - 세전 CAGR
  - 세후 CAGR
  - MDD
  - 최대 원금 손실률

## 8) 2차 실험 거버넌스
- 모든 실험 run 기록(`run_id`, 파라미터, 데이터/코드 버전)
- 채택 조건:
  - OOS 유지율 >= 70%
  - 리스크 지표 악화 과도하지 않음
- v1(동치)와 v2+(개선안) 결과를 절대 혼합하지 않음

## 9) 3차 운영 대시보드 필수 블록 (텔레그램 화면 매핑)
샘플 이미지: `9551781008_192159903_f42a776fa8b229fbdd2105c6cd411d85.webp`

필수 영역:
1. 알림 헤더(날짜/알림유형)
2. 포지션 변경(현재/교체/매수·매도 비율/손익/로스컷)
3. 사유(reason)
4. 시장 데이터 요약(TQQQ/QQQ/SPY/변동성/RSI/환율)
5. 운영 로그(run_id/버전/전송 성공 여부)

## 10) 3안(데이터레이크/오케스트레이터) 전환 트리거
아래 2개 이상 충족 시 전환 검토:
- 전략 3개 이상 동시 운영
- 일일 배치 30분 이상 + 재시도 빈번
- 수동 운영 개입 주 2회 이상
- 동시 사용자 3명 이상

## 11) 현재 비범위(Non-goals)
- 브로커 API 자동 주문
- 초단타/인트라데이 체결엔진
- 마이크로서비스 분리/대규모 오케스트레이션 즉시 도입

## 12) 리스크 및 대응
- 데이터 이질성(yfinance vs stooq): source별 정합 검사 + mismatch 리포트
- 과최적화: OOS 70% 게이트 + 스트레스 구간 분리
- 알림 오발송/중복: 멱등키(run_id+date+signal) 적용
